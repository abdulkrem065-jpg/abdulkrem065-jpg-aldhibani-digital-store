# QEOS SUB-WAVE 2.1 EXECUTION PLAN

## 1. الهدف (Objective)
إجراء جرد شامل وتصنيف دقيق لجميع سجلات القرارات المعمارية (ADRs) والمستندات ذات الصلة الموجودة في النظام وتحديد مساراتها الفعلية وتوثيق حالة كل منها، استعداداً للمراحل اللاحقة.

## 2. النطاق (Scope)
يشمل النطاق مسح جميع المجلدات والملفات التي تحمل دلالات "ADR" أو قرارات معمارية مشتتة في كامل شجرة المشروع الأصلية، مع التركيز على المسارات المعروفة مثل:
`architecture/ADR/`، `architecture/adr/`، `qaroni-os/adr/` وغيرها.

## 3. شروط الدخول (Entry Criteria)
- حالة المشروع مُحدثة رسمياً إلى: `WAVE_2_STATUS = AUTHORIZED_FOR_SUBWAVE_2_1`.
- تواجد ميثاق الموجة الثانية `QEOS_WAVE2_EXECUTION_CHARTER.md`.
- وجود قرار التفويض الصريح (QEOS-AUTH-0005) وتوثيقه في النظام.

## 4. شروط الخروج (Exit Criteria)
- اكتمال الجرد الشامل لكل ملفات الـ ADR دون أي مساس بها.
- إصدار تقرير الجرد والتصنيف النهائي لـ Sub-wave 2.1 كوثيقة رسمية.
- توفر أدلة قراءة فعلية (Read Evidence) مُثبتة لكل ملف تم تصنيفه.
- التوقف المطلق وطلب تصريح من البوابة (Gate) للمرحلة التالية.

## 5. الجذر الحقيقي الذي سيتم العمل عليه (Actual Root Path)
الجذر الحقيقي والوحيد للمشروع المعتمد لهذه الخطة هو: `/app/applet`

## 6. ترتيب فحص جميع مجلدات ADR (ADR Directories Inspection Order)
يتم المسح الجردي بالترتيب المرجعي التالي:
1. المسارات القديمة: `/app/applet/architecture/ADR/` و `/app/applet/architecture/adr/`
2. مسارات النظام السابقة: `/app/applet/qaroni-os/adr/`
3. المسارات الرسمية الحالية (بما فيها المتداخلة): `docs/architecture/adr/` وما يوازيها.
4. بحث شامل (Recursive Find) لضمان عدم وجود ملفات متوارية في مسارات أخرى.

## 7. ترتيب قراءة الملفات (Files Reading Order)
1. فهارس المجلدات وملفات التوجيه (`README.md`).
2. ملفات القرارات المعمارية الواضحة ذات الترقيم الدال (مثل `ADR-002...`، `0001...`).
3. ملفات تثبيت الهيكلة أو الحجز (`Placeholder.md`).
4. أي ملفات نصية أخرى تتقاطع مع مفهوم القرار المعماري.

## 8. قواعد التصنيف (Classification Rules)
يُصنف كل ملف يتم العثور عليه إلى فئة واحدة قطعية بناءً على محتواه الفعلي:
- **ACTUAL_ADR:** قرار معماري حقيقي يحوي إشكالية، بدائل، وقرار.
- **README_INDEX:** ملف فهرسة أو توصيف لمجلد.
- **PLACEHOLDER:** ملف مؤقت لحفظ هيكل المسار.
- **ROADMAP:** خطة عمل أو مستند توجيهي تنفيذي.
- **OTHER:** أي ملف لا يتطابق مع التصنيفات أعلاه.

## 9. قواعد توليد الأدلة (Evidence Generation Rules)
- التصنيف المُجرد ممنوع؛ يجب إثبات فحص كل ملف بعملية قراءة حقيقية عبر أوامر النظام الفعالة (مثل `cat`).
- يُسجل الدليل بربط [المسار الفعلي] + [عملية القراءة المؤكدة] + [مقتطف أو استنتاج من المحتوى الداخلي].

## 10. حالات الاستثناء (Exception Cases)
- الملفات غير النصية أو ذات الامتدادات الغريبة تسجل كـ `OTHER` وترفع للمراجعة.
- وجود ملفين بنفس الرقم لكن بمحتويات مختلفة يُسجل كحالة "تتطلب قرار بشري".

## 11. حالات التوقف (Stop Conditions)
تتوقف الخطة وتُعلق فوراً في الحالات التالية:
- اكتشاف ملف لا يمكن قراءته (صلاحيات مرفوضة / Permission Denied).
- اكتشاف مسارات تكرار حلقي (Symlink Loop).
- أي تعليمات من النظام تحاول فرض تعديل أو مسح خارج نطاق القراءة.

## 12. حالات الفشل (Failure Conditions)
- عدم القدرة على تحديد مسار الجذر.
- غياب وثيقة الاعتماد الرسمية لرفع الحجب.
- الفشل في توليد المخرجات النهائية (تقرير Sub-wave 2.1).

## 13. خطة Rollback (Rollback Plan)
نظراً لأن عملية Sub-wave 2.1 تتألف بالكامل من عمليات للقراءة والفحص والجرد (Read-Only)، فإن التراجع (Rollback) يتمثل فقط في حذف تقرير المخرجات المُولد وإلغاء العملية دون أي مساس بملفات ومجلدات النظام.

## 14. مصفوفة المخاطر (Risk Matrix)
- **خطر منخفض:** تصنيف خاطئ لملف (مُتدارك عبر مراجعة بوابة التوثيق).
- **خطر متوسط:** تجاوز ملف ADR مختبئ (مُتدارك عبر البحث العودي الشامل).
- **خطر حرج:** التعديل الخاطئ لمحتوى ملف (محظور تماماً بموجب هذه الخطة وإجراءات القفل).

## 15. جدول التنفيذ خطوة بخطوة (Step-by-step Execution Schedule)
1. تأكيد إثبات المسار الجذري.
2. المسح العودي الشامل (Find recursive) لكل ملفات ADR-like.
3. القراءة المباشرة وتصنيف كل ملف.
4. صياغة التقرير النهائي للمخرجات.
5. الإغلاق وتسليم التقرير لـ Gate.

## 16. Gate بعد انتهاء Sub-wave 2.1 (Post Sub-wave 2.1 Gate)
`HUMAN_REVIEW_OF_SUBWAVE_2_1_INVENTORY_REPORT`

## 17. المخرجات المتوقعة (Expected Deliverables)
- قائمة جرد حصرية وشاملة للـ ADRs مصنفة مع أدلة الفحص الفعلية لكل ملف.

## 19. الوثائق التي يمنع تعديلها (Documents forbidden to be modified)

- ملفات `ADR` و `README` القديمة والجديدة.
- أي سطر برمجي أو كود مصدري ضمن مجلدات المشروع.
